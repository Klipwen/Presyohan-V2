Presyohan Mobile - Add Multiple Items UI/UX Upgrade Implementation Plan
Generated: 2026-06-06

Objective
Upgrade the current Presyohan Mobile Add Multiple Items workflow into a multi-entry import flow that supports:

1. Simple Mode
- Manual item entry.
- Multiple categories.
- Multiple items per category.
- Editable fields per item: Item Name, Unit/Size, Price, Description optional.
- Add Item button.
- Delete Item button.
- Category selector.
- Total item counter.
- Review button.

2. Fast Mode
- Bulk text entry mode inside Add Multiple Items.
- Toggle between Simple Mode and Fast Mode.
- User can paste multiple items at once.
- Parser extracts category, item name, description, price, and unit.
- Parser supports examples such as:
  [FRUITS]
  Apple (Red) - PHP 120.00 | Kilo
  Banana - PHP 90.00 | Dozen

  [DRINKS]
  Coke 1.5L - PHP 95.00 | Bottle
  Water - PHP 20.00 | Bottle
- Parsed items become draft item objects.
- Review screen validates malformed entries and duplicates before save.

3. Import Prices Feature
- Add a visible Import Prices button from the Add Multiple Items screen.
- Import modal supports:
  - Import Excel File.
  - Paste Raw Text.
- Excel import supports .xlsx selection, upload/parse, then Review.
- Paste Raw Text supports supplier price lists, then Review.

4. Review & Validation Screen
- Final step before save for Simple, Fast, Excel, and Paste Raw Text flows.
- Groups items by category.
- Shows NEW, UPDATE, DUPLICATE, and INVALID status.
- Shows summary counts.
- Allows Edit Items.
- Saves valid NEW/UPDATE items, skips DUPLICATE items, and blocks DONE when invalid items remain unless product decides to allow Save Valid Items Only.

5. Edit Imported Items
- Review screen includes Edit Items.
- Edit returns to Add Multiple Items in Simple Mode.
- Imported/parsed results are pre-populated and editable.
- Changes persist until saved or discarded.

6. Draft Protection
- Leaving Add Multiple Items with unsaved draft data shows:
  Title: Discard Draft Items?
  Message: The items you have entered have not been saved and will be permanently lost.
  Buttons: Discard All, Cancel


Complete Feature-Gap Analysis

Current implementation summary:
- AddItemActivity launches AddMultipleItemsActivity from the Add Multiple Items button.
- AddMultipleItemsActivity is currently a raw text parser screen, not a manual multi-category editor.
- activity_add_multiple_items.xml contains a ViewFlipper with two states:
  - Input state: store info, example format, raw EditText, Preview Items button.
  - Preview state: summary text, RecyclerView preview, Edit Text button, Save Items button.
- AddMultipleItemsParser parses raw pasted text into ParsedCategory and ParsedItem.
- ItemImportModels defines:
  - ItemStatus: NEW, UPDATE, DUPLICATE, ERROR_NO_PRICE, ERROR_INVALID_FORMAT, ERROR_NO_CATEGORY.
  - ParsedItem: name, description, unit, price, status.
  - ParsedCategory: name, mutable item list.
- ImportManager saves ParsedCategory/ParsedItem through SupabaseImportRepository.
- SupabaseImportRepository ensures categories with add_category and saves products with direct update or add_product RPC.
- Existing Import Prices dialogs in HomeActivity, StoreActivity, and ManageStoreActivity use dialog_import_prices.xml.
- Existing Import Prices behavior:
  - Excel option only shows "Excel import is available on the web version."
  - Copy via Paste-Code opens CopyPricesActivity.
  - CopyPricesActivity is a store-to-store paste-code copy flow, not supplier raw text or Excel import.

Current architectural issues to address:
- AddMultipleItemsActivity owns UI, parsing, validation-ish logic, and save orchestration in one Activity.
- Preview is not a true validation/review workflow.
- Parsed item statuses are created during parsing, but validation should happen as a separate step before Review.
- Existing product matching is not stable enough because parser checks lowercased names but save lookup uses exact name.
- There is no draft/session model, so edit imported items cannot safely round-trip through Review.
- No Excel import reader exists in mobile.
- No draft discard protection.


Phase 2 - Database Changes

Required database schema changes:
- No mandatory database schema change is required for the requested UI/UX and workflow.
- Existing products and categories tables already support:
  - category
  - product name
  - description
  - price
  - unit

Recommended optional database/API improvements:
1. Add server-side bulk validation RPC.
- Optional RPC name: validate_import_items.
- Purpose:
  - Accept store_id and normalized candidate product keys.
  - Return existing product IDs and duplicate ambiguity.
  - Reduce mobile round trips for large imports.
- Benefit:
  - More reliable UPDATE detection.
  - Less client-side filtering for large files.

2. Add server-side bulk upsert RPC.
- Optional RPC name: import_products_bulk.
- Purpose:
  - Accept validated NEW/UPDATE items.
  - Create categories.
  - Insert new products.
  - Update existing products by product_id.
  - Return save summary.
- Benefit:
  - Avoid one network call per item.
  - Better transaction behavior.

Phase 4 - Data Flow, State Management, Parsing, and Validation

Recommended state management:
- Introduce AddMultipleItemsViewModel.
- State should include:
  - current mode
  - draft session
  - category list
  - existing products cache
  - import modal state
  - validation summary
- Use LiveData or StateFlow depending on current project preference.
- Since the current app uses XML/AppCompat and lifecycleScope, LiveData is acceptable; StateFlow is cleaner if the team is comfortable with it.

Draft persistence:
- Store draft session JSON in cacheDir/import_drafts/{sessionId}.json.
- Save on every meaningful draft mutation or before navigation to Review.
- Load session by sessionId when Review opens or Edit Items returns.
- Clear session after successful Done or Discard All.

Parser strategy:
- Keep AddMultipleItemsParser as the base parser because it already supports many supplier/export formats.
- Refactor parser output from ParsedCategory/ParsedItem into a richer ParseResult:
  - categories
  - invalid lines
  - warnings
  - source row numbers
  - original lines
- Preserve current parser support:
  - bracket headers
  - plain category headers
  - bullet items
  - descriptions in parentheses
  - price after dash/long dash/colon/greater-than
  - pipe unit suffix
  - PRICELIST/date/header skipping
- Add parser improvements:
  - Do not classify every no-digit text line as a category if it appears after item context and has item-like separators.
  - Validate empty product name after stripping description/separators.
  - Detect negative price explicitly.
  - Preserve invalid lines as DraftItem with INVALID status and error reason.
  - Normalize Unicode separators and line breaks.
  - Accept "PHP", "P", and peso symbol variants when practical.
  - Support long dash and simple hyphen consistently.

Excel parsing strategy:
- Add an Android-compatible XLSX reader dependency.
- Prefer a lightweight reader compatible with Android over Apache POI unless size/method-count compatibility is verified.
- Parse first worksheet by default.
- Support header-based columns:
  - category
  - item/product/name
  - description
  - price
  - unit/size
- Support fallback row pattern if file mirrors exported Presyohan pricelists.
- Convert each row into DraftItem with sourceRowNumber.
- Validation handles missing category/name/price/unit.

Validation strategy:
- Create ImportValidationUseCase.
- Input:
  - storeId
  - draft categories/items
  - existing products fetched from Supabase
- Output:
  - validated draft categories/items
  - summary counts
  - warnings
- Validation rules:
  - Product name required.
  - Category required.
  - Price required and numeric.
  - Price must not be negative.
  - Unit required if product decides unit is required; otherwise default to "1pc" for parser imports.
  - Invalid format rows become INVALID.
  - Duplicate within import list becomes DUPLICATE.
  - Existing product in database becomes UPDATE.
  - New product becomes NEW.
- Duplicate key recommendation:
  - Normalize with trim, collapse whitespace, lowercase.
  - Business decision needed for match key:
    - Safer update key: store + normalized product name + normalized description.
    - Current app mostly uses product name only.
  - Recommended: use name + description for UPDATE matching because CopyPricesActivity already avoids merging different descriptions.
- If multiple existing DB products match the same normalized key:
  - Mark item as DUPLICATE or INVALID with DUPLICATE_IN_DATABASE_AMBIGUOUS.
  - Require user edit before Done.

Save strategy:
- Update ImportManager to save by validated status and productId.
- NEW:
  - Ensure category.
  - Insert product with add_product.
- UPDATE:
  - Ensure category.
  - Update existing product by productId, not exact name.
- DUPLICATE:
  - Skip.
- INVALID:
  - Block Done unless Save Valid Items Only is approved.
- For large imports, consider a bulk RPC after the first client implementation works.


Phase 5 - Review Screen

Purpose:
- Final confirmation point for all Add Multiple Items import paths.
- Review parsed/imported/manual items before saving.
- Surface duplicates and invalid rows clearly.
- Allow correction through Edit Items.

Entry points:
- Simple Mode -> Review.
- Fast Mode -> Parse Data -> Review.
- Excel Import -> Parse Excel -> Review.
- Paste Raw Text -> Parse Text -> Review.

Screen implementation recommendation:
- Create ReviewImportActivity.
- Register it in AndroidManifest.
- It receives:
  - storeId
  - storeName
  - draftSessionId
- It loads draft data from local draft session cache.
- It runs ImportValidationUseCase before rendering.
- It writes validation results back to the same draft session.

Display layout:
- Header:
  - Back icon.
  - Title: Review.
- Warning/summary strip:
  - Found X Duplicates!
  - Found Y Invalid items!
  - Edit Items button.
- Grouped category list:
  FRUITS
  Apple [NEW]
  Description: Red Apple
  PHP 120.00 / Kilo

  Banana [DUPLICATE]
  Description: Cavendish
  PHP 90.00 / Kilo

  Orange [INVALID]
  Description: Imported
  PHP 150.00 / Kilo
- Bottom summary:
  - New Items: N
  - Update Items: N
  - Duplicate Items: N
  - Invalid Items: N
  - Total Categories: N
  - Total Items: N
  - DONE button.

DONE behavior:
- If invalid count is 0:
  - Save NEW items.
  - Update UPDATE items.
  - Skip DUPLICATE items.
  - Show success toast/snackbar:
    - Successfully imported N items.
  - Clear draft session.
  - Return to HomeActivity or previous store screen.
- If invalid count is greater than 0:
  - Disable DONE.
  - Show:
    - X invalid items require attention before saving.
- Optional product decision:
  - Add Save Valid Items Only button.
  - If enabled, it saves NEW/UPDATE only and leaves invalid/duplicate skipped.

Edit Items behavior:
- User taps Edit Items.
- ReviewImportActivity saves current draft session.
- Navigate to AddMultipleItemsActivity with:
  - mode = SIMPLE
  - draftSessionId
  - storeId
  - storeName
- AddMultipleItemsActivity opens Simple Mode and pre-populates all items.
- User edits category/product/description/unit/price.
- User taps Review again.
- Validation runs again.

Back behavior from Review:
- If draft has unsaved changes, show the same discard confirmation or return to Add Multiple Items depending on UX decision.
- Recommended:
  - Back from Review returns to Add Multiple Items Simple Mode with draft preserved.
  - Back from Add Multiple Items triggers Discard Draft Items if dirty.

Phase 8 - Risks and Edge Cases

Duplicate products:
- Current app has inconsistent duplicate matching between parser and save.
- Risk: case-only product differences can be marked UPDATE but inserted as new.
- Mitigation: validation should store existingProductId and save UPDATE by ID.

Empty categories:
- Parser can create invalid/no-category groups.
- Manual UI can create empty category sections if user selects category but adds no items.
- Mitigation: do not count empty categories; show INVALID for items missing category; remove empty category sections on Review.

Invalid prices:
- Inputs may include commas, peso symbol, PHP text, spaces, zero, negative values, or malformed decimals.
- Current parser can accidentally treat negative price as a separator plus positive number.
- Mitigation: parse price text explicitly and flag NEGATIVE_PRICE/INVALID_PRICE.

Large Excel files:
- Large imports can exceed memory, take long to parse, or cause too many per-item network calls.
- Mitigation: stream rows if library supports it, cap preview size warnings, show loading overlay, and consider bulk RPC.

Parser failures:
- Supplier lists vary widely in punctuation, headers, bullets, and line breaks.
- Mitigation: preserve original lines, show INVALID rows in Review, and allow Edit Items to fix them.

Unsaved drafts:
- Users can accidentally leave after adding many rows.
- Mitigation: dirty-state tracking and Discard Draft Items modal.

Android Intent size:
- Passing full import data between activities can fail for large imports.
- Mitigation: pass only draftSessionId and store draft JSON in cacheDir.

Excel dependency compatibility:
- Apache POI can be heavy on Android.
- Mitigation: choose a lightweight Android-compatible reader and test release build method count and startup cost.

Category normalization:
- Existing add_category uppercases category names.
- UI may show category names typed by user before server normalization.
- Mitigation: normalize category display in validation or after category ensure response.

Database ambiguity:
- Existing stores may already contain duplicate product names.
- Mitigation: if more than one existing product matches, mark imported item as DUPLICATE or INVALID until user edits.

Network/RLS failures:
- Import save depends on Supabase RPC/table permissions.
- Mitigation: keep existing add_category/add_product paths, surface failures in result, and avoid deleting draft until save succeeds.

Mode switching data loss:
- Switching Simple/Fast can wipe draft or raw text if not designed carefully.
- Mitigation: keep independent mode state and ask before replacing non-empty drafts.

Business rule ambiguity: DUPLICATE vs UPDATE
- Requirement mentions database duplicates but also says existing products should be UPDATE.
- Recommended interpretation:
  - Existing product with one clear DB match = UPDATE.
  - Duplicate within imported list = DUPLICATE.
  - Multiple existing DB matches = DUPLICATE or INVALID requiring user action.

Recommended implementation approach:
- Build the new draft/session/validation architecture first.
- Then wire Simple Mode and Review.
- Then reuse the same pipeline for Fast Mode. 
- Then add Import Prices modal.
- Then add Excel import.
- This order gives the team a working manual Review flow early, while all later import types reuse the same validation and save path.

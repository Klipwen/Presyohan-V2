PRESYOHAN MOBILE - ENTRY AND ONBOARDING SCREENS FEATURE GAP ANALYSIS AND IMPLEMENTATION PLAN

Prepared for: Presyohan Mobile
Scope: Native Android Entry/Auth Screens, Password Recovery Flow, Signup Verification Flow, Entry Modals, and the new Onboarding Activity flow.

================================================================================
Objective
================================================================================
Refresh the Presyohan Mobile entry screens and introduce the new multi-step Onboarding Activity flow. The onboarding experience will guide first-time, record-free users through personalized paths (Store Owner/Staff vs. Customer) and trigger appropriate post-launch actions (e.g., auto-opening Join/Create Store or Suki dialogs) based on their choices, while standardizing entry inputs (using email/name and excluding usernames) and confirmation dialogs (using the reusable template).

================================================================================
Deliverables
================================================================================

--------------------------------------------------------------------------------
Phase 1 – Current System Analysis
--------------------------------------------------------------------------------
1. Current Android Entry Architecture:
   - SplashActivity, LoginActivity, SignupActivity, and VerifyEmailActivity manage authentication.
   - Authentication operations are centralized in SupabaseAuthService.
   - Entry screens use XML-based layouts and AppCompatActivity.
   
2. Onboarding Activity (None implemented yet):
   - Currently, verified users are routed directly to StoreActivity or the last active screen.
   - There is no guided wizard or role selection screen immediately after verification.
   
3. Login Screen (activity_login.xml):
   - Current: Contains fields for email and password.
   - Request: Disregard username (use email on the login screen), which is already aligned with current login implementation, but we must ensure no username references remain in visual labels or variables.
   
4. Signup Screen (activity_signup.xml):
   - Current: Uses name, email, password, and confirm password fields.
   - Request: Disregard username (use name on sign up screen). We will ensure the signup form requests display "Name" and not "Username".

5. Confirmation Dialogs:
   - Current: ReusableDialogHelper displays confirmation messages but is not consistently used for exiting setup or cancelling password resets.
   - Request: Standardize both "Exit Setup?" and "Cancel Password Reset?" dialogs using the existing `ReusableDialogHelper` (which utilizes `dialog_reusable_template.xml`).

--------------------------------------------------------------------------------
Phase 2 – Database Changes
--------------------------------------------------------------------------------
No new columns are required in Supabase database tables for onboarding or login flows.
We will query existing tables to dynamically check if the user is a fresh user with no existing records:
1. Query table `store_members` filter by `user_id == current_user.id` to see if they are a member/owner of any store.
2. Query table `suki_relationships` filter by `user_id == current_user.id` to see if they have established suki relationships.

Phase 4 – State Management
--------------------------------------------------------------------------------
Onboarding Flow State (stored locally inside OnboardingActivity viewmodel/state):
- `selectedRole`: Enum (STORE | CUSTOMER | NONE)
- `storeAction`: Enum (CREATE | JOIN | NONE)
- `sukiPreference`: Enum (YES | NO | NONE)
- `currentStep`: Int (1, 2, or 3)

Post-Launch Pending Action (stored in SharedPreferences):
- Key: `onboarding_completed` (Boolean) - Set to true when the user launches the app from the onboarding success screen.
- Key: `onboarding_action_pending` (String) - Used to execute actions upon landing on the home screens:
  - `"create_store"`: Automatically launch Create Store Dialog in StoreActivity.
  - `"join_store"`: Automatically launch Join Store Dialog in StoreActivity.
  - `"add_suki"`: Automatically launch Add Suki Dialog in CustomerHomeActivity.
  - `"select_presyohan"`: No dialog, but routes to SelectPresyohanActivity immediately.

Phase 6 – Navigation Flow
--------------------------------------------------------------------------------
1. Splash Screen Route Guard:
   - Check if user session is active.
   - If logged in:
     - Query Supabase `store_members` where `user_id == current_user.id`.
     - Query Supabase `suki_relationships` where `user_id == current_user.id`.
     - If both query results are empty AND SharedPreferences key `onboarding_completed` is false:
       - Route to `OnboardingActivity`.
     - Else:
       - Route to last saved screen (StoreActivity or CustomerHomeActivity) as normal.
   - If not logged in:
     - Route to `LoginActivity`.

2. Onboarding Wizard Transitions:
   - Page 1 (Choose Role) -> Click Next ->
     - If Role == STORE: Go to Page 2 (Store Path: Create vs Join)
     - If Role == CUSTOMER: Go to Page 2 (Customer Path: Yes vs No suki)
   - Page 2 -> Click Next ->
     - If Role == STORE: Go to Page 3 (Store Success Screen)
     - If Role == CUSTOMER: Go to Page 3 (Customer Success Screen)
   - Page 3 -> Click LAUNCH APP ->
     - Mark `onboarding_completed = true` in SharedPreferences.
     - Set SharedPreferences key `onboarding_action_pending` based on selection.
     - Route user:
       - If Role == STORE: Route to `StoreActivity`.
       - If Role == CUSTOMER and sukiPreference == NO: Route to `SelectPresyohanActivity`.
       - If Role == CUSTOMER and sukiPreference == YES: Route to `CustomerHomeActivity`.

3. Post-Launch Modal Triggers:
   - StoreActivity Lands:
     - If `onboarding_action_pending` == `"create_store"`: open `CreateStoreDialogHelper.showCreateStoreDialog(this)`, then clear the pending action key.
     - If `onboarding_action_pending` == `"join_store"`: open `JoinStoreDialogHelper.showJoinStoreDialog(this)`, then clear the pending action key.
   - CustomerHomeActivity Lands:
     - If `onboarding_action_pending` == `"add_suki"`: open the suking tindahan dialog (`showAddSukiDialog()`), then clear the pending action key.

Phase 8 – Risks and Edge Cases
--------------------------------------------------------------------------------
1. Bypassing Onboarding on App Reinstall:
   - Risk: If a user has existing records in the database (e.g., they already own a store or follow a suki) but reinstalls the app, local SharedPreferences flags are wiped.
   - Mitigation: The Splash router queries the remote Supabase tables (`store_members` and `suki_relationships`) on launch to verify history. If records exist, onboarding is automatically skipped.
2. Network Interruption During Login Router Checks:
   - Risk: Slow internet blocks splash routing because database queries are required to check store/suki history.
   - Mitigation: Default to cached session/SharedPreferences states if queries fail, or display a retry overlay.
3. Uncompleted Setup Sessions:
   - Risk: User starts onboarding, minimizes/closes the app, then re-enters.
   - Mitigation: Since the `onboarding_completed` flag remains false and no database entries exist, they will safely resume at the onboarding screen.

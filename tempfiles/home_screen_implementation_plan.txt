================================================================================
PRESYSOHAN MOBILE APPLICATION - HOME SCREEN REDESIGN IMPLEMENTATION PLAN
================================================================================

OBJECTIVE
--------------------------------------------------------------------------------
The objective of this document is to detail the architectural, database, and UI/UX
changes required to transition the Presyohan Mobile application's Home Screen
from its current flat grid representation to the new grouped category layout.
This redesign addresses three distinct user roles (Owner, Manager, and Sales
Staff), introducing a categorized card structure, a persistent docked bottom
sheet (with rounded top corners) containing search inputs and expandable action
controls, inline context actions (edit/delete overlays), pull-to-refresh
capabilities, and role-appropriate dialog options, while preparing the database
schema (store and product visibility columns) and product layout indicators for 
future public store and suki partnership visibility.

NOTE: This plan focuses strictly on the Store/Merchant side app (Owners, Managers, 
and Sales Staff). Partnered customer (Suki) viewing screens, suki login roles, 
and customer-facing layouts are out of scope and excluded from this implementation.

Furthermore, this plan incorporates three core premium enhancements:
1. Shimmer Skeleton Loading: Replacing the central spinner with animated 
   placeholder cards.
2. Minimal Empty State Indicator: Introducing a clean center-aligned visual 
   directly inside the main layout for empty listings or search results.
3. Optimistic UI Updates: Instantly updating the UI on modifications and 
   rolling back only upon network failure to eliminate wait states.

PHASE 1 - CURRENT SYSTEM ANALYSIS
--------------------------------------------------------------------------------
The current home screen feature is implemented across the following files:
1. com.presyohan.app.HomeActivity.kt:
   - Manages store details loading, role validation, notification count badge.
   - Triggers the 'get_store_products' RPC on Supabase to fetch store products.
   - Implements search via an EditText (toggleable visibility) and scroll hiding.
   - Houses custom Dialog popups for store menu options (dialog_store_menu.xml) 
     and product management (dialog_manage_item.xml).
   - Category filtering uses a standard Android Spinner hidden behind a label 
     (layoutPricelistTrigger) which pops open a dropdown list.
2. activity_home.xml (Layout):
   - Uses a DrawerLayout with a NavigationView for the side menu.
   - Header contains the app logo, notifications icon, and profile avatar.
   - Sub-header banner (bg_header_yellow) displays the store name, branch 
     name, back button, and a three-dot options icon (btnStoreOptions).
   - Center body displays the product list using a RecyclerView styled with a
     GridLayoutManager (2 columns, spanCount=2).
   - Bottom right features an ImageButton (searchItemButton) that acts as the 
     primary floating action to open the search bar.
3. com.presyohan.app.adapter.ProductAdapter.kt:
   - A standard RecyclerView.Adapter displaying products flatly.
   - Binds name, description, price, and volume into a simple vertical card 
     layout (item_product.xml).
   - Long-presses on list items trigger a manage item dialog callback.

CRITICAL GAPS TO RESOLVE:
- Grid layout vs Grouped card layout: Current code displays items in a flat 
  2-column grid. The new design requires products grouped into separate cards 
  representing each category, arranged as a vertical single-column list.
- Options Menu: Current options menu is identical for all roles (via 
  dialog_store_menu.xml). The new layout dictates that:
  - Owners get a floating card menu (dialog_owner_menu.xml) displaying the store
    icon, name, type, branch, and two action icons: Invite Member and Store 
    Settings on the right.
  - Managers and Sales Staff get a detailed "Store Details" panel containing 
    store ID, creation dates, public status, invite code, item counts, and a
    breakdown of member roles, alongside a prominent "LEAVE" store action.
- Add and Search triggers: The current add button is located in the upper 
  control bar. The new design features a persistent bottom sheet containing 
  search inputs, and a floating search button (searchItemButton) at the 
  bottom-right.
- Bottom Sheet and FAB Interaction Logic:
  * When the bottom sheet is collapsed or hidden, the floating search button 
    (searchItemButton) appears on the bottom right. Tapping the search button 
    slides the bottom sheet UP.
  * When the bottom sheet is UP (visible/collapsed), the floating search button 
    is hidden.
  * When the user scrolls down the pricelist, the bottom sheet slides down and 
    hides itself completely off-screen, giving the user a full-screen view. The 
    floating search button reappears.
- Dynamic Pricelist Constraint (Anti-Overlapping):
  * When the bottom sheet is hidden, the pricelist (RecyclerView) must stretch 
    all the way to the bottom of the screen with no extra empty space.
  * When the bottom sheet slides up, the product list must be pushed upward 
    (by dynamically adjusting its bottom margin/padding to match the visible 
    height of the sliding sheet) so that no list items are overlapped or 
    hidden behind the bottom sheet.
- Product Editing: Current system displays a modal dialog (dialog_manage_item) 
  for actions. The new design uses an in-place overlay card over the individual 
  product item. This overlay contains Pencil (Edit) and Trash (Delete) icons,
  revealed ONLY upon long-pressing the item (normal clicks/taps are ignored).
- Loading Feedback: Current implementation displays a full-screen blocking progress 
  overlay. The redesign replaces this with a non-blocking skeleton shimmer layout.
- Pull-to-Refresh: Current app has no pull-to-refresh mechanism; loading items 
  is only done on launching or manually updating. The new design will include a
  minimal, professional SwipeRefreshLayout tied to the Presyohan palette.
- Empty State Feedback: The current app displays no visual fallback when no 
  products are available or when searches fail. A minimal centered visual (icon and 
  text) will be embedded directly in the main layout.
- Store & Item Visibility (Store/Merchant Management side):
  * Store members (Owner, Manager, Sales Staff) must be able to view, manage, 
    and identify public vs private items. A mini globe icon is placed beside the 
    product name on the pricelist row if `product.is_public == true`.
  * Owners and Managers will be able to toggle individual products to public or 
    private status during product creation or editing, and toggle the store itself 
    to public or private.


PHASE 2 - DATABASE CHANGES
--------------------------------------------------------------------------------
To support the store visibility levels, public item toggles, and partnered customer 
relationships, the following schema modifications are proposed:

-- Migration File: 20260611_000000_add_public_and_suki_entities.sql

-- 1. Add 'is_public' flag to stores (default is private)
ALTER TABLE public.stores 
ADD COLUMN IF NOT EXISTS is_public BOOLEAN NOT NULL DEFAULT FALSE;

-- 2. Add 'is_public' flag to products (default is private)
-- Toggling this allows select items to be visible to Sukis when the store is public
ALTER TABLE public.products 
ADD COLUMN IF NOT EXISTS is_public BOOLEAN NOT NULL DEFAULT FALSE;

-- 3. Define Suki partnerships (Customer to Store relationship)
CREATE TABLE IF NOT EXISTS public.suki_relationships (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES public.app_users(id) ON DELETE CASCADE,
    store_id UUID NOT NULL REFERENCES public.stores(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    status TEXT NOT NULL DEFAULT 'active',
    CONSTRAINT suki_user_store_unique UNIQUE (user_id, store_id)
);

-- 4. Indexes for optimization
CREATE INDEX IF NOT EXISTS suki_relationships_user_idx ON public.suki_relationships (user_id);
CREATE INDEX IF NOT EXISTS products_public_idx ON public.products (store_id, is_public);

-- 5. Enable Row Level Security (RLS) and define Policies
ALTER TABLE public.suki_relationships ENABLE ROW LEVEL SECURITY;

-- Suki RLS Policies
DROP POLICY IF EXISTS suki_select_own ON public.suki_relationships;
CREATE POLICY suki_select_own ON public.suki_relationships
    FOR SELECT TO authenticated
    USING (user_id = auth.uid());

DROP POLICY IF EXISTS suki_insert_own ON public.suki_relationships;
CREATE POLICY suki_insert_own ON public.suki_relationships
    FOR INSERT TO authenticated
    WITH CHECK (user_id = auth.uid());

PHASE 4 - STATE MANAGEMENT
--------------------------------------------------------------------------------
To facilitate role-based features, interactive layouts, and optimistic updates, 
we will introduce the following states inside HomeActivity.kt:

1. UI States:
   - activeOverlayProductId (String?): Tracks which product currently has its 
     Edit/Delete action overlay visible. Only one product can show this overlay 
     at a time. It is activated ONLY via a Long-click on a product item.
     Clicking outside or long-clicking a different item clears or shifts this state.
     Normal clicks/taps on a product item do not open this overlay.
   - searchSheetState (Enum): Tracks whether the docked bottom sheet is:
     * HIDDEN (completely off screen, FAB visible)
     * COLLAPSED (showing search bar and handle, FAB hidden)
     * EXPANDED (showing bottom quick action drawer, FAB hidden)
   - currentQuery (String): Stores the search input for filtering.
   - isRefreshingPricelist (Boolean): Tracks SwipeRefreshLayout status.
   - isListLoading (Boolean): Toggles visibility of the ShimmerFrameLayout vs 
     the parent RecyclerView.

2. Data States & Optimistic UI Cache:
   - userRole (String?): "owner", "manager", or "employee".
   - storeIsPublic (Boolean): Caches whether the current store is public or private.
   - groupedProducts (Map<String, List<Product>>): Replaces the flat products list.
   - cachedBackupProducts (Map<String, List<Product>>?): Holds the backup state 
     of the product list immediately before executing a write command.
   - storeStats (StoreMetadata): Cache stores stats (member counts, categories 
     count) fetched asynchronously to populate the Store Details dialog instantly.

OPTIMISTIC UI UPDATE FLOW:
1. User triggers an operation (e.g. Deletes product with ID 'X' from category 'Y').
2. Backup Current State: Copy `groupedProducts` directly into `cachedBackupProducts`.
3. Apply Mutation Locally: Remove the product from `groupedProducts` list and notify 
   the adapter to refresh immediately. The UI updates instantly.
4. Execute Network Request: Trigger the Supabase database write call in a coroutine.
5. Handle Results:
   - If Success: Clear `cachedBackupProducts` and complete.
   - If Failure: Restore `groupedProducts = cachedBackupProducts`, notify adapter to 
     reinstate the item, and alert the user with a Toast notification.


PHASE 6 - NAVIGATION FLOW
--------------------------------------------------------------------------------
The application navigation from the Home Screen behaves as follows:

1. Core Screen Loading:
   Home Screen Launches -> Start Shimmer Loader -> Fetches Store Status & User Role ->
   Always load all products (both public and private) -> Renders outer category cards -> 
   Hides Shimmer -> Shows yellow FAB. Bottom sheet starts in HIDDEN status.
   * If database returns empty -> Show empty state visual container.
   * If `product.is_public` is true, show the inline globe icon next to the product name.

2. Slide Down / Pull-to-Refresh Flow:
   Slide down from the top of the category cards list -> Triggers SwipeRefreshLayout
   indicator -> Calls Supabase get_store_products RPC -> Hides indicator on callback.

3. Bottom Sheet and FAB Toggle Flow:
   - Tap Yellow FAB -> Bottom Sheet slides up from bottom (COLLAPSED status) ->
     Yellow FAB is animated/hidden.
   - Scroll Down Pricelist -> Bottom Sheet slides down completely (HIDDEN status) ->
     Yellow FAB is animated/visible -> Pricelist bottom margin is removed to fill screen.
   - Slide/Drag Orange Handle Up -> Bottom Sheet expands (EXPANDED status).
   - Tap Outside / Swipe Down -> Bottom Sheet goes back to COLLAPSED or HIDDEN state.

4. Dynamic Push Up (Anti-Overlap) Flow:
   - As the bottom sheet slides up, a BottomSheetCallback listener intercepts the 
     slide offset.
   - The list layout's bottom margin is updated in real-time to match the visible 
     pixel height of the sliding bottom sheet, pushing the list items up dynamically.
   - When the sheet slides completely down, the bottom margin reverts to 0dp, 
     stretching the list to the full height of the screen.

5. Search Flow:
   Type search query in bottom sheet's Edit Text -> Child RecyclerViews instantly 
   filter to display matches.
   * If search result is empty -> Show empty state visual container.

6. Quick Action Sheet Flow (Owners & Managers only):
   In EXPANDED bottom sheet status:
   - Tap "+ Add Item" -> Launch AddItemActivity (updates list on result).
   - Tap "Manage Prices" -> Launch ManageItemsActivity.
   - Tap "Manage Categories" -> Launch ManageCategoryActivity.

7. Options Menu Flow:
   Tap "..." in banner:
   - If User is Owner -> Open dialog_owner_menu.xml -> User has options to click 
     Invite Member or Store Settings. (Store settings allows toggling store `is_public`).
   - If User is Manager/Staff -> Open dialog_store_details.xml -> Leave Store or Close.

8. Product Action Overlay:
   Long-press Product item -> Show inline Edit/Delete overlay ->
   - Tap Pencil icon -> Launch EditItemActivity (prefilled with product data, 
     including "Public item" toggle switch).
   - Tap Trash icon -> Perform Optimistic UI Delete -> Delete Confirmation Dialog -> 
     Trigger Supabase delete call.
   - Tap background -> Clear activeOverlayProductId and dismiss overlay.
   - (Note: Normal taps/clicks do not trigger this overlay, avoiding mis-clicks).

PHASE 8 - RISKS AND EDGE CASES
--------------------------------------------------------------------------------
1. Nested RecyclerView Scroll lag:
   - Risk: Having nested scrollable inputs can lead to stuttering frame rates.
   - Mitigation: Set nested scrolling to false on the child recyclerviews 
     (childProductRecyclerView.isNestedScrollingEnabled = false). Use 
     RecyclerView.RecycledViewPool shared across child adapters if necessary.
     (make sure that no lag occurs when scrolling)
2. Soft Keyboard UI distortion:
   - Risk: Toggling the search edit text will raise the keyboard, potentially 
     crushing list items or pushing the bottom sheet off screen.
   - Mitigation: Use windowSoftInputMode="adjustResize" in AndroidManifest.xml 
     and ensure the bottom sheet layout height/gravity matches requirements.
3. RLS / Role Sync delays:
   - Risk: If a user is demoted or leaves, cached roles might allow unauthorized 
     database actions, causing RLS exceptions.
   - Mitigation: Ensure appropriate error handling (try-catch) on Supabase CRUD 
     actions, showing clear toasts and refreshing local views immediately.
4. SwipeRefresh Conflict with Scroll:
   - Risk: Dragging down to collapse the bottom sheet or scroll the list could 
     trigger SwipeRefresh prematurely.
   - Mitigation: Enable/disable the SwipeRefreshLayout.isEnabled flag dynamically 
     based on the scroll position of the RecyclerView (only enable when scrollY == 0).
5. Optimistic Action Race Conditions:
   - Risk: User performs multiple rapid actions (e.g. deletes 2 products in rapid 
     succession). If the first fails, the rollback might restore the second 
     deleted product incorrectly or clash with pending requests.
   - Mitigation: Use a FIFO request queue for mutations, or disable other actions 
     temporarily during the short asynchronous execution window if conflict risk is high.
